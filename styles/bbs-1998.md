# BBS 1998 Style Guide

Apply this as a presentation layer. Do not bake it into the source templates.

## Visual Rules

- Use ASCII dividers.
- Use bracketed labels.
- Use compact section headers.
- Use monospace-friendly formatting.
- Avoid emoji and modern decorative symbols.
- Prefer status-line language.

## Header Pattern

```txt
:::[ {{template_title}} ]::::::::::::::::::::::::::::::::::::
```

## Section Pattern

```txt
[ {{section_label}} ]
{{section_body}}
```

## Footer Pattern

```txt
::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::
[STATUS: {{status}}] [CONFIDENCE: {{confidence}}] [END]
```

## Batch Patterns

Use these when the content is a multi-prompt batch (structure:
`templates/prompts/prompts.prompt-batch.standard.md`). The boundaries are the
point: a reader must never miss a prompt or mistake where one stops.

### Batch banner

Open the whole batch with a banner that carries the prompt count, then list the
manifest under it:

```txt
:::[ PROMPT BATCH · {{batch_title}} · {{prompt_count}} PROMPTS ]::::::::::::::
```

### Numbered item banner

Open each prompt with a full-width banner. Always show `n OF N` so a missing
prompt is obvious from the count alone:

```txt
=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=
  PROMPT {{n}} OF {{N}}  ·  {{prompt_title}}
  {{repo}}  ·  issue #{{issue}}  ·  lane {{lane_id}}
=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=
```

### Sub-item label

A sub-part stays inside its prompt and is labelled with the parent number plus a
letter — never its own banner:

```txt
[ {{n}}{{letter}} · {{subpart_label}} ]
```

### Item closer

Close each prompt so the gap between two prompts cannot be misread as part of
either one:

```txt
=-=-=-=-=-=-=-=-=  END PROMPT {{n}} OF {{N}}  =-=-=-=-=-=-=-=-=
```

## A Note On Color

Color was considered for separating prompts and rejected as non-portable. Terminal
ANSI escape codes do not survive Markdown rendering, and Markdown has no portable
text color (GitHub strips it). These prompts are read in a terminal **and** pasted
into issues and PRs, so any color would vanish on one path or the other. The
full-width `=-=-=-=` banner is the BBS-era substitute for color: it renders
identically in a terminal, a Markdown preview, and a GitHub paste. Lean on structure
and the `n OF N` count, not color.
